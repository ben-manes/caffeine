DESIGN PRIORS (intentional patterns documented in .claude/docs/design-decisions.md and cross_model_audit_results.md — DO NOT flag a diff that matches one of these):
  - Lossy/best-effort: read buffer drops, approximate frequency counts, eventual consistency, transient negative weightedSize, size() is documented as estimate, expiration is amortized not instant, no debug-mode assertions
  - Refresh is lock-free and best-effort: refreshIfNeeded reads writeTime/getKey/getValue/isAlive without synchronized(node); one extra loader call per rare interleaving is acceptable; discardRefresh over-aggressively invalidates concurrent refresh generations (correct for linearizability); writeTime ABA check is legacy extra-safety, not load-bearing for refresh
  - Async listeners fire only on resolved values: AsyncRemovalListener.onRemoval uses thenAccept and silently drops exceptional/cancelled futures; AsyncEvictionListener uses Async.getIfReady and skips in-flight/exceptional; analogous to "Async REPLACED only fires if the old value resolves"
  - LoadingCache.refresh and refreshAll Javadoc "exception logged and swallowed" refers to the cache MAPPING (failed reload leaves previous value in place, not removed); the returned future itself can still complete exceptionally — that's the caller-observable signal
  - AsyncCache.synchronous().asMap() query/mutation split: containsKey/get/iteration/containsValue treat in-flight as absent (logical); KeySet.remove/removeAll/removeIf/retainAll and EntryIterator.remove operate physically on the raw delegate map. Inherent sync-over-async tradeoff
  - LocalAsyncCache.put(k, null-or-failed-future) calls unconditional cache().remove(key) — by design, last-writer-wins
  - Weight=0 is a pinning feature (not eligible for eviction)
  - Weak/soft references: cleared refs are eventually drained by drainKeyReferences/drainValueReferences; cleared-ref equals returning true for two cleared refs is harmless because CHM indexes by stored hash; remove on a cleared key is harmless because lookup misses
  - accessTime uses opaque write (no CAS) by design; backward movement = benign early expiration. writeTime uses plain write under synchronized(node)
  - notifyEviction is called BEFORE user code; catch-commit-rethrow handles user-thrown exceptions in mappingFunction/weigher/expiry. Same-instance return from compute/merge skips setValue but still updates weight/accessTime/variableTime/writeTime (intentional)
  - ASYNC_EXPIRY ~220 years and MAXIMUM_EXPIRY ~150 years are durations, not absolute times. hasExpired uses (now - variableTime) >= 0 signed subtraction (overflow-safe)
  - CHM bin blocking on slow loaders is not a Caffeine bug; no close() by design (cache is GC-eligible when unreferenced)
  - asMap() iteration is not a cache read (does not update access times or frequency)
  - WriteThroughEntry.setValue is not fully atomic — documented per Map contract
  - Serialization proxy hardening (readObject/readObjectNoData) is defense-in-depth, not a bug if absent — cache is not designed for adversarial deserialization
